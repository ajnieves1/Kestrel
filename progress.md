## B. Conventions for every task

- Style rules from `design.md` section 4 apply to every file, including SDF,
  YAML, and shell. No hyphenated words outside upstream tool syntax, no em
  dashes, one plain comment line above every function, first line of every
  Python file is a one line purpose comment.
- Python nodes follow this shape: a `Node` subclass, a module level
  `# Start the node and spin` comment above `def main():`, `main` does
  `rclpy.init`, construct, spin (executor per A12), destroy, shutdown.
  `entry_points` in `setup.py` map `node_name = kestrel.module:main`.
- Parameters: every tunable declared with `declare_parameter` and read once in
  `__init__`, defaults must match `config/kestrel_params.yaml`.
- Service client pattern: `wait_for_service(timeout_sec=1.0)` in a loop that
  logs a waiting message, then `call_async`, then wait on the future with a
  deadline. Never `spin_until_future_complete` inside a callback.
- Logging: `self.get_logger().info/warn/error`. Guard trips and aborts log at
  error level.
- Commits: one commit per task minimum, subject under 60 chars, imperative
  mood, no Claude or AI mention anywhere per design.md section 1 rule 6.

## TASK 5: telemetry monitor node

Goal: first ROS Project node, proves rclpy plumbing and QoS.

Files: `src/kestrel/kestrel/telemetry_monitor.py`, entry point in `setup.py`.

Implement design.md contract with these exact members:

```python
# Watch vehicle state and battery and log a one line summary each second
class TelemetryMonitor(Node):
    # Subscribe to mavros state, battery, and local position
    def __init__(self):
    # Store the latest vehicle state message
    def on_state(self, state_message):
    # Store the latest battery message
    def on_battery(self, battery_message):
    # Store the latest local position message
    def on_pose(self, pose_message):
    # Log mode, armed flag, battery percent, and altitude
    def log_summary(self):
```

Design points:

- Topics: `/mavros/state` (`mavros_msgs/State`), `/mavros/battery`
  (`sensor_msgs/BatteryState`), `/mavros/local_position/pose`
  (`geometry_msgs/PoseStamped`). All three with `qos_profile_sensor_data`

- A 1.0 s timer calls `log_summary`.
- Summary line format:
  `mode=GUIDED armed=True battery=87% altitude=4.98m`. Battery percent is
  `battery_message.percentage * 100` rounded to an int. Altitude is
  `pose_message.pose.position.z` with two decimals. Any field not yet received

- Single threaded spin is fine here.

`flight_commander.py` members:

```python
# Offer takeoff, goto, and land services that wrap MAVROS control of ArduCopter
class FlightCommander(Node):
    # Create service servers, MAVROS clients, subscriptions, and state
    def __init__(self):
    # Handle /kestrel/cmd/takeoff by arming and climbing to the target altitude
    def handle_takeoff(self, request, response):
    # Call a MAVROS service and wait for the reply with a deadline
    def call_service_blocking(self, client, request, timeout_seconds):
```

Takeoff sequence, in order, each step fails the service with a clear
`response.message` on timeout:

1. Wait until `/mavros/state` reports `connected` (30 s deadline).
2. Wait until one `/mavros/home_position/home` message has arrived, this is
   the EKF gate (A14), 90 s deadline.
3. Set mode `GUIDED` through `/mavros/set_mode` (`mavros_msgs/srv/SetMode`,
   `custom_mode='GUIDED'`), confirm via the state subscription, 10 s deadline.
4. Arm through `/mavros/cmd/arming` (`mavros_msgs/srv/CommandBool`), up to 5
   attempts 3 s apart, ArduPilot rejects until checks pass.
5. Call `/mavros/cmd/takeoff` (`mavros_msgs/srv/CommandTOL`) with only
   `altitude` set.
6. Poll `/mavros/local_position/pose` until `z` is within 0.5 m of the target,
   30 s deadline.

Threading (A12): all subscriptions, service servers, and clients share one
`ReentrantCallbackGroup`, `main` spins a `MultiThreadedExecutor`. Inside
`handle_takeoff`, waiting means a loop of `time.sleep(0.1)` checks against
stored subscription state, and futures from `call_async` are awaited in the
same loop style with a deadline, never `spin_until_future_complete`.

Verify, three container shells (launch, telemetry monitor, commander), then:

```bash

ros2 service call /kestrel/cmd/takeoff kestrel_msgs/srv/Takeoff "{altitude: 5.0}"
```

Done when the call returns `success: true` and telemetry shows about 5 m.
Paste both.

## TASK 7: goto and land

Goal: full basic flight loop.

Files: `flight_commander.py`, `scripts/fly_square.sh`. `GotoLocal.srv` exists
from task 6:

```
float32 north
float32 east
float32 altitude
float32 yaw_deg
---
bool success
string message
```

`yaw_deg` is compass heading, 0 north, clockwise positive (A4).

New members on `FlightCommander`:

```python
    # Handle /kestrel/cmd/goto by streaming setpoints until arrival or timeout
    def handle_goto(self, request, response):
    # Handle /kestrel/cmd/land by calling MAVROS land and waiting for disarm
    def handle_land(self, request, response):
    # Convert north, east, altitude, and compass yaw to an ENU PoseStamped
    def build_enu_setpoint(self, north, east, altitude, yaw_deg):
```

Design points:

- The one and only frame conversion in the project lives in
  `build_enu_setpoint`, with one comment above it explaining that MAVROS local
  frame is ENU while our services speak north and east: ENU `x = east`,
  `y = north`, `z = altitude`. Yaw per A24: ENU yaw radians
  `= radians(90 - yaw_deg)`, quaternion `z = sin(yaw / 2)`,
  `w = cos(yaw / 2)`, `x = y = 0.0`.
- `handle_goto`: refuse with a message if not armed or not in GUIDED. Publish
  the setpoint to `/mavros/setpoint_position/local` at 10 Hz from a
  `time.sleep(0.1)` loop inside the handler (the executor is multithreaded,
  other callbacks keep running). Arrival check each iteration: 3D distance
  between current pose and target below 0.5 m. Deadline 60 s, then
  `success false` with the remaining distance in the message. Stop publishing
  after arrival, GUIDED holds position.
- `handle_land`: call `/mavros/cmd/land` (`CommandTOL`, zeros), then wait for
  `/mavros/state` `armed` to go false, 60 s deadline.

`scripts/fly_square.sh`: bash, `set -euo pipefail`, six `ros2 service call`
commands in sequence: takeoff 5, goto the four corners of a 10 m square
(`(10, 0)`, `(10, 10)`, `(0, 10)`, `(0, 0)`, altitude 5, `yaw_deg 0`), land.

Verify: run the square with SITL, MAVROS, telemetry monitor, and commander up:

```bash

```

Done when all six calls return success and telemetry shows disarm at the end.

Design, exact:

- States are the strings `IDLE`, `TAKEOFF`, `SURVEY`, `INVESTIGATE`,
  `RESUME`, `RETURN`, `LANDED`. Published on `/kestrel/mission_state`
  (`std_msgs/String`) by a 1 Hz timer and on every change.
- Threading per A12: subscriptions and timers on a `MultiThreadedExecutor`,
  the mission sequence runs in one `threading.Thread` started from `__init__`
  when `auto_start` is true.
- `run_mission` flow:
  1. `IDLE`: wait for `/kestrel/cmd/takeoff`, `/kestrel/cmd/goto`,
     `/kestrel/cmd/land` services to exist.
  2. `TAKEOFF`: call takeoff with altitude equal to the survey start altitude
     (compute with the planner rule, `max(3.0, climb_step)`). Failure means
     log error and go to `RETURN`.
  3. `SURVEY`: build the helix once with `build_survey_path`. Fly waypoints
     by index. After each arrival, if the defect queue is not empty, remember
     the current index and go `INVESTIGATE`.
  4. `INVESTIGATE`: pop one event, build the orbit with `build_orbit_path`
     around a `Waypoint` made from the event world position, fly all orbit
     waypoints, then `RESUME`. Drain repeats if more events queued.
  5. `RESUME`: continue the helix at the remembered index, state goes back to
     `SURVEY`.
  6. After the last helix waypoint, `RETURN`: goto
     `(0.0, 0.0, return_altitude, 0.0)`, then call land.
  7. `LANDED` after land succeeds. Log a summary: waypoint count flown,
     defect count investigated, wall duration. Thread ends, node keeps
     spinning.
- Any goto failure: log error, skip to `RETURN` (the safety guard exists, but
  the director still gives up cleanly).
- The queue is a `collections.deque`, `on_defect_event` appends. No dedupe
  here, the detector already fires once per marker.
- All movement goes through commander services, never raw setpoints
  (design.md section 1 rule 7).

through), plus nodes `telemetry_monitor`, `flight_commander`, `safety_guard`,
`defect_detector`, `mission_director`, `report_writer` (exists after task 14,
